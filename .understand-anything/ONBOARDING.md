## What is QAI?

QAI is a three-service application for quiz-based learning with an AI study coach:

```
┌─────────────────────────────────────────────────────────────────┐
│  Browser                                                         │
│  Next.js (:3000) — Pages Router, Zustand state, Tailwind CSS    │
└────────┬───────────────────────────┬────────────────────────────┘
         │                           │
         ▼                           ▼
┌────────────────────┐     ┌──────────────────────────────────┐
│ Spring Boot (:8080)│     │ AI Study Coach (:8000)            │
│ Quiz CRUD, scoring │◄────│ WebSocket chat, LLM, SM-2 algo   │
│ Firestore access   │     │ Progress tracking, scheduler      │
└────────┬───────────┘     └──────────────────────────────────┘
         │
         ▼
   Google Firestore
```

**Core flows:**
1. Users create and play quizzes (frontend ↔ Spring Boot)
2. AI Coach chats in real-time via WebSocket (frontend ↔ AI Coach)
3. After quiz completion, a webhook triggers spaced repetition scheduling (Spring Boot → AI Coach)
4. Background scheduler sends review notifications (AI Coach → Spring Boot → frontend)

## Step 3: Frontend Architecture

**Entry:** `frontend/src/pages/_app.js`

This is where everything starts. It:
- Sets up Firebase auth listener (`onIdTokenChanged`)
- Syncs auth tokens to HTTP-only cookies for BFF API routes
- Renders global dialogs: `AuthForm`, `PlayForm`, `CreateQuizRoomForm`
- Mounts the `StudyCoachWidget` (floating chat bubble)

### Pages (Next.js Pages Router)

| Page | Path | What it does |
|------|------|-------------|
| Home | `/` | Category grid, game mode selection |
| Play | `/play` | Active quiz gameplay with timer, questions, scoring |
| Create | `/create` | Quiz creation form with question editor |
| Profile | `/profile` | User stats, quiz history, leaderboard |
| Coach | `/coach` | AI coaching dashboard (progress, reviews, materials) |
| Chat | `/chat` | Full-page AI chat transcript |

### State Management (Zustand)

All stores are composed into one via `useBoundStore.js`:

| Store | File | Manages |
|-------|------|---------|
| `useAuth` | `store/useAuth.js` | User object, auth readiness |
| `useQuestions` | `store/useQuestions.js` | Current game questions, answers, score |
| `useWildcards` | `store/useWildcards.js` | Power-ups (50/50, skip, etc.) |
| `useChat` | `store/useChat.js` | WebSocket connection, message history |
| `useCoach` | `store/useCoach.js` | Coach dashboard state |
| `useCreate` | `store/useCreate.js` | Quiz creation form state |
| `useQueries` | `store/useQueries.js` | Dialog visibility flags |

### API Layer (BFF Pattern)

Frontend API routes in `pages/api/` act as a Backend-For-Frontend. They:
1. Validate the Firebase token cookie
2. Forward requests to Spring Boot or AI Coach
3. Return sanitized responses

Key routes: `/api/quiz/*`, `/api/question/*`, `/api/take/*`, `/api/coach/*`, `/api/auth/*`.

## Step 4: Spring Boot Backend

**Entry:** `QuizzAiOnlineApplication.java`

### Controllers (REST API)

| Controller | Endpoints | Purpose |
|-----------|-----------|---------|
| `QuizController` | CRUD quizzes | Create, list, search quizzes |
| `QuestionController` | CRUD questions | Add/update questions per quiz |
| `TakeQuizController` | Start/end attempts | Track quiz sessions and scores |
| `TakeQuestionController` | Save answers | Record per-question responses |
| `ReviewScheduleController` | CRUD schedules | SM-2 review dates from AI Coach |
| `NotificationController` | CRUD notifications | Review reminders, milestones |
| `UserController` | User ops | Profile management |

### Services

Each controller delegates to a service that handles business logic and Firestore access:
- `QuizService` — quiz CRUD + pagination + category filtering
- `TakeQuizService` — scoring, attempt management, `WebhookService` trigger
- `WebhookService` — fires POST to AI Coach when a quiz ends
- `ReviewScheduleService` — CRUD for spaced repetition schedules
- `NotificationService` — CRUD for in-app notifications

### Data Layer

- **Models:** `Quiz`, `Question`, `TakeQuiz`, `TakeQuestion`, `ReviewSchedule`, `Notification`, `User`
- **DTOs:** Request/Response pairs for each model (e.g., `QuizCreationRequestDto` / `QuizResponseDto`)
- **Firestore:** Direct SDK access (no JPA/Hibernate — document DB)
- **Config:** `FirebaseConfiguration` initializes Firestore client, `SecurityConfig` sets CORS + auth

## Step 5: AI Study Coach

**Entry:** `ai-study-coach/server/main.py`

### Architecture

```
WebSocket ─→ ws/endpoint.py ─→ router.py ─→ capabilities/lite_orchestrator.py
                                                    │
                              ┌──────────────────────┼──────────────────────┐
                              ▼                      ▼                      ▼
                   capabilities/chat.py   capabilities/quiz.py   capabilities/solve.py
                              │                      │                      │
                              └──────────┬───────────┘                      │
                                         ▼                                  │
                              agent/coach.py (agentic loop)                 │
                                         │                                  │
                              ┌──────────┼──────────────────────────────────┘
                              ▼          ▼
                       tools/registry.py ─→ tools/*.py
                              │
                    ┌─────────┼─────────────┐
                    ▼         ▼             ▼
              tools/rag.py  tools/web_search.py  tools/quiz_history.py
```

### Capabilities (Intent Routing)

The `lite_orchestrator` classifies user messages and routes to:

| Capability | Trigger | What it does |
|-----------|---------|-------------|
| `chat` | General conversation | Streams LLM response with RAG context |
| `quiz` | "Generate questions about..." | Creates questions via LLM, saves to Spring Boot |
| `solve` | "Solve this step by step" | Breaks problems into guided steps |
| `agentic` | Complex multi-tool requests | Full tool-use loop (reason → act → observe) |

### Tools

Registered in `tools/registry.py`:

| Tool | Purpose |
|------|---------|
| `rag` | Retrieves relevant study materials from Supabase vector store |
| `web_search` | Searches the internet for current information |
| `quiz_history` | Fetches user's past quiz performance |
| `recommend` | Suggests next study topics based on weaknesses |
| `reason` | Internal chain-of-thought reasoning |

### Learning Algorithms

| Module | File | Algorithm |
|--------|------|-----------|
| Spaced Repetition | `learning/spaced_repetition.py` | SM-2 (interval, easiness, repetitions) |
| Progress Tracking | `learning/progress.py` | Mastery scores, velocity, streaks, trends |
| Weakness Detection | `learning/weakness.py` | Category-level performance analysis |
| Scheduler | `scheduler/scheduler.py` | APScheduler checks due reviews hourly |

### LLM Providers

| Provider | File | Use case |
|----------|------|----------|
| LM Studio | `llm/lm_studio.py` | Local inference (default, port 1234) |
| DeepSeek | `llm/deepseek.py` | Cloud API (full tier, higher quality) |
| External | `llm/external.py` | Generic OpenAI-compatible API |

## Step 6: Key Data Flows

### Flow A: User Plays a Quiz

```
1. PlayForm.jsx opens → user selects quiz
2. frontend/pages/play → loads questions from Spring Boot
3. QuestionSlider.jsx renders questions one by one
4. User answers → checkAnswer API validates
5. Quiz ends → TakeQuizController.end() computes score
6. TakeQuizService triggers WebhookService
```

### Flow B: Webhook → AI Coach → Review Scheduling

```
1. WebhookService.java POSTs to AI Coach /webhook/quiz-completed
2. webhook.py stores event, runs SM-2 algorithm
3. SM-2 computes next review date based on performance
4. AI Coach PUTs schedule to Spring Boot /review-schedule
5. AI Coach POSTs notification to Spring Boot /notifications
```

### Flow C: WebSocket Chat

```
1. StudyCoachWidget connects WebSocket to :8000/ws
2. User sends message → ws/endpoint.py receives
3. router.py → lite_orchestrator classifies intent
4. Capability streams LLM response tokens back
5. If agentic: tool calls execute, results feed back to LLM
```

### Flow D: Background Scheduler

```
1. APScheduler fires every hour
2. Checks review_schedule for due items
3. Creates notification via Spring Boot API
4. NotificationBell.jsx polls for new notifications
```

## Step 7: Conventions & Patterns

### Frontend
- **Styling:** Tailwind CSS utility classes (no CSS modules)
- **State:** Zustand with slices composed in `useBoundStore`
- **Auth:** Firebase + HTTP-only cookie for SSR
- **API calls:** Always go through `/pages/api/*` BFF routes (never call Spring Boot directly from browser)
- **Components:** Feature-grouped in `components/{Feature}/`

### Spring Boot
- **Architecture:** Controller → Service → Firestore (no repository layer)
- **DTOs:** Always use DTOs at API boundary, never expose models directly
- **IDs:** Generated via `IdUtil.java` (NanoID-style)
- **Timestamps:** Custom `TimestampDeserializer` for Firestore Timestamp ↔ Java
- **Config:** `application.properties` + `@Value` injection

### AI Coach
- **Async everywhere:** All HTTP calls use `httpx.AsyncClient`
- **Streaming:** LLM responses stream token-by-token over WebSocket
- **Config:** Pydantic `Settings` class with `.env` file support
- **Capabilities pattern:** Each capability is a class with `handle()` method
- **Tool registration:** Decorator-based `@register_tool` in `tools/registry.py`

## Step 10: Common Tasks

### Add a new REST endpoint

1. Create/update DTO in `spring-backend/src/.../dto/`
2. Add method to relevant Service class
3. Add `@PostMapping`/`@GetMapping` to Controller
4. Add BFF route in `frontend/src/pages/api/`
5. Call from frontend component

### Add a new AI Coach tool

1. Create `ai-study-coach/server/tools/my_tool.py`
2. Use `@register_tool` decorator with name, description, parameters
3. Import in `tools/__init__.py`
4. The agentic capability will auto-discover it

### Add a new frontend page

1. Create `frontend/src/pages/mypage/index.js`
2. Next.js auto-routes it to `/mypage`
3. Use `useBoundStore` for state access
4. Wrap with `RequireAuth` if login required

### Add a new Zustand store slice

1. Create `frontend/src/store/useMySlice.js`
2. Export a function that takes `set, get`
3. Import and spread in `useBoundStore.js`

## Architecture Decisions (ADRs)

| Decision | Rationale |
|----------|-----------|
| Pages Router (not App Router) | Simpler, team familiarity, stable SSR |
| Zustand (not Redux) | Minimal boilerplate, works with Next.js SSR |
| Firestore (not Postgres) | Serverless, real-time subscriptions, no schema migrations |
| BFF pattern | Security (hide Spring Boot from browser), auth cookie handling |
| WebSocket for chat | Token-by-token streaming, bidirectional communication |
| SM-2 algorithm | Well-studied, proven for spaced repetition, simple to implement |
| Separate AI service | Different runtime (Python), independent scaling, LLM isolation |
