# Chapter 4: System Design — Part 2: Data Models and API Contracts

## 4.8 Database Design

### 4.8.1 Firestore Collections (Spring Boot Domain)

Firestore, a document-oriented NoSQL database, stores the primary application data. The schema uses top-level collections with document references rather than relational joins.

**Collection: `quiz`**

| Field | Type | Description |
|-------|------|-------------|
| id | string (auto) | Document ID |
| host_id | string | User ID of the quiz creator |
| title | string | Quiz title |
| description | string | Optional description |
| status | enum | ACTIVE, INACTIVE |
| categories | array[enum] | List of category enums |
| start_time | timestamp | Scheduled start (optional) |
| end_time | timestamp | Scheduled end (optional) |
| availability | string | Access control (optional) |

**Collection: `question`**

| Field | Type | Description |
|-------|------|-------------|
| id | string (auto) | Document ID |
| quiz_id | string | Reference to parent quiz |
| content | string | Question text |
| answers | array[string] | Exactly 4 answer options |
| correct_answer | string | The correct answer text |

**Collection: `take_quiz`**

| Field | Type | Description |
|-------|------|-------------|
| id | string (auto) | Document ID |
| quiz_id | string | Reference to quiz taken |
| player_id | string | User who took the quiz |
| player_name | string | Display name of player |
| score | string | Format "correct/total" (e.g., "3/5") |
| start_time | timestamp | When attempt started |
| end_time | timestamp | When attempt completed |

**Collection: `take_question`**

| Field | Type | Description |
|-------|------|-------------|
| id | string (auto) | Document ID |
| take_id | string | Reference to parent attempt |
| question_id | string | Reference to question |
| answer | string | User's chosen answer |
| check_answer | string | "1"/"2" for correct, "-1" for incorrect |

**Collection: `review_schedules`**

| Field | Type | Description |
|-------|------|-------------|
| id | string (auto) | Document ID |
| user_id | string | User this schedule belongs to |
| category | string | Category being reviewed |
| easiness | double | SM-2 easiness factor (≥1.3, default 2.5) |
| interval_days | double | Days until next review |
| repetitions | int | Consecutive successful recalls |
| next_review | timestamp | When review is due |
| last_reviewed | timestamp | Last review date |
| last_score | string | Score from last review |
| updated_at | timestamp | Last update timestamp |

**Collection: `notifications`**

| Field | Type | Description |
|-------|------|-------------|
| id | string (auto) | Document ID |
| user_id | string | Recipient user |
| type | string | REVIEW_DUE, MILESTONE, etc. |
| title | string | Notification title |
| message | string | Notification body |
| read | boolean | Whether user has seen it |
| metadata | map | Additional data (category, quiz_id) |
| created_at | timestamp | Creation timestamp |

**Collection: `users/{uid}/documents`** (Frontend-managed)

| Field | Type | Description |
|-------|------|-------------|
| id | string | Document ID (generated by frontend) |
| name | string | Original filename |
| status | string | processing, indexed, failed |
| ragStatus | string | pending, indexed, failed |
| ragError | string | Error message if RAG failed |
| ragDocumentId | string | ID in Supabase for deletion |
| uploadedAt | timestamp | Upload timestamp |
| questions | array | Generated questions (optional) |
| ragChunks | number | Count of RAG chunks stored |

### 4.8.2 Supabase Tables (RAG Storage)

**Table: `documents`**

| Column | Type | Description |
|--------|------|-------------|
| id | uuid (PK) | Auto-generated |
| content | text | Chunk text content |
| metadata | jsonb | Source file info, chunk index |
| embedding | vector(768) | nomic-embed-text-v1.5 embedding |
| kb_id | text | Knowledge base ID (= user_id) |

Index: `ivfflat` on `embedding` column using cosine distance for approximate nearest-neighbor search.

The `match_documents` RPC function performs similarity search:
```sql
CREATE FUNCTION match_documents(
  query_embedding vector(768),
  match_count int DEFAULT 5,
  filter_kb_id text DEFAULT NULL
) RETURNS TABLE(id uuid, content text, metadata jsonb, similarity float)
```

### 4.8.3 SQLite (AI Coach Local State)

### 4.9.2 AI Coach REST API

| Method | Path | Description |
|--------|------|-------------|
| POST | /ingest | Upload and index document |
| DELETE | /ingest/{document_id} | Delete document chunks |
| POST | /generate/from-topics | Generate questions from topic |
| POST | /generate/from-file | Generate questions from uploaded file |
| POST | /webhook/quiz-completed | Process quiz completion event |
| GET | /health | Health check |

**Example: POST /ingest**

Request (multipart/form-data):
```
file: <binary PDF/TXT/MD>
kb_id: "user_uid_here"
```

Response (200):
```json
{
  "document_id": "abc123",
  "chunks_stored": 12,
  "message": "Document indexed successfully"
}
```

Error (400 — image-only PDF):
```json
{
  "detail": "PDF contains no extractable text (image-only). Please use a text-based PDF."
}
```

**Example: POST /generate/from-topics**

Request:
```json
{
  "topics": ["photosynthesis"],
  "num_questions": 5,
  "difficulty": "medium",
  "document_name": "biology-chapter-3.pdf",
  "user_id": "Hk1UMM0yRWbymqtxZNZjp6ZaUMs1"
}
```

Response:
```json
{
  "questions": [
    {
      "content": "What is the primary pigment in photosynthesis?",
      "answers": ["Chlorophyll", "Melanin", "Hemoglobin", "Carotene"],
      "correct_answer": "Chlorophyll"
    }
  ]
}
```

## 4.10 Category Taxonomy

Categories are defined as a Java enum shared across quiz and analytics:

```java
public enum Category {
    SCIENCE, HISTORY, GEOGRAPHY, LITERATURE, TECHNOLOGY,
    SPORTS, ENTERTAINMENT, MATH, ART, SPACE, GENERAL_CULTURE
}
```

While the `Quiz` model stores categories as a `List<Category>`, the spaced repetition system uses only the first category (`quiz.getCategories().get(0)`) for per-category analytics. This effectively treats each quiz as having a primary category for scheduling purposes.
